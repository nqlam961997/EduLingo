## Context

EduLingo's roleplay chat is built on three providers (Gemini, OpenAI, Local) behind an `AiService` interface. The default is `ai.provider=local`, which routes through a FastAPI bridge to Ollama running `gemma2:9b`. The current chat path:

- Frontend ([ChatView.vue](../../../frontend-vue/src/views/ChatView.vue)) re-sends the full history on every turn.
- Backend ([ChatService.java](../../../backend-spring/src/main/java/com/edulingo/service/ChatService.java)) builds a fresh system prompt from a hard-coded template and **flattens** the history into a single user message of the form `"Learner: ... \n You: ..."`.
- The persona is two strings (name, role) from `TopicDto.java`.
- The scenario text is generated by the same weak model and stored only inside the message history.
- No session state, no scratchpad, no eval harness. The only tests on the chat path are 5 mock-model plumbing tests in [ChatServiceTest.java](../../../backend-spring/src/test/java/com/edulingo/service/ChatServiceTest.java) that never assert on prompt content.

Empirical failure modes (per user report and prompt audit): the AI forgets earlier turns, drifts off persona, misreads intent, and produces canned replies. The audit identified four leak points (flattened history, no scenario state, no per-turn intent reasoning, output-format pressure on a small model) and ten per-topic prompt defects (article gender, role/action mismatch, recursive frame for school, asymmetric topics misusing suggestions, etc.).

This change is the first structured spec in this codebase (`openspec/specs/` is empty). It also lands during the in-flight `docker-deployment` change, which it must not regress.

## Goals / Non-Goals

**Goals:**
- Eliminate the four context-leak architectural causes (flattened history, no state, no intent reasoning, output-format pressure) on the default provider (`gemma2:9b` via local Ollama).
- Produce structurally consistent persona behavior across the 12 topics by replacing the thin "name + role" persona with a typed **persona card** and a **3-type topic taxonomy** (transactional / asymmetric / free-form).
- Make chat behavior **observable and testable** through a three-layer eval harness, so prompt and design changes can no longer ship on vibes.
- Make scenario text **deterministic** by replacing model-generated scenarios with curated scenario seeds per topic.
- Stay binary-compatible with the existing public HTTP API (no breaking change to clients other than an additive `sessionId`).

**Non-Goals:**
- Cross-session memory ("last time we talked about hotel check-inâ€¦"). Deferred.
- Refresh resilience / multi-device session migration. The minimum `chat_session` row supports it but the frontend doesn't restore mid-session on refresh in v1.
- Streaming-UX fix where JSON characters leak through to the user during stream. Acknowledged but out of scope; a separate change will address output shape.
- TTS, Live2D, microphone, or any audio pipeline change.
- Picture-mode (`PictureService`) is unchanged. The persona-card and scratchpad concepts could later extend to it; this change explicitly does not.
- An RL/feedback loop or A/B testing of personas.
- Localization of persona-card content; `explain_vi` behavior is preserved as-is.

## Decisions

### D1. Scratchpad sourced by **second-pass extraction** on a small model (Option B), not model-emitted (A) or hybrid delta (C).

**Why:** The single biggest issue audited is too many simultaneous constraints on a 9B model. Option A adds another. Option B is the only choice that *reduces* main-reply constraints. Local Ollama makes the extra call essentially free; the small extraction model (`qwen2.5:1.5b` candidate) is fast enough to land before the typical learner can compose the next turn. Option C remains an attractive fallback if extraction latency becomes a problem in production.

**Alternative considered:** Model-emitted state (A) â€” rejected because it worsens the audited #1 problem. Hybrid delta (C) â€” held in reserve; we'll have everything needed to swap to it without changing the chat-side contract if extraction proves too slow.

### D2. **Three-type topic taxonomy** with per-topic overrides, not 12 bespoke schemas.

**Why:** The stress test in exploration showed that schema shape clusters cleanly into three: slot machine (transactional), phase machine (asymmetric), rhythm beat (free-form). Twelve schemas duplicate logic; one schema fits none of them. Per-topic overrides handle the awkward cases (doctor's `SUGGEST=OFF` despite being slot-shaped; school's recursive-frame guard).

**Alternative considered:** Flat per-topic schemas â€” rejected for duplication. Single universal schema â€” rejected because asymmetric and free-form genuinely need different structural primitives.

### D3. Persona cards live in **`TopicDto.java`** as a structured record, not in separate YAML/Markdown files.

**Why:** 12 topics Ă— ~30 lines is bounded. Java records give compile-time safety on persona field completeness. The "recompile to edit" cost is offset by these being domain-stable (Maria's vibe doesn't iterate weekly). Authoring tools (LSP, refactor, find-references) work natively.

**Alternative considered:** YAML files under `personas/` â€” rejected for v1 because the iteration speed gain is small and the test/validation surface becomes harder. We'd revisit if persona authoring becomes a contributor-facing activity.

### D4. **New `chat_session` JPA table** for session-level state, not in-memory or Redis.

**Why:** The project already uses Spring Data JPA; no new infra. Sessions survive backend restart. `chat_session` carries the scratchpad JSON (jsonb), `topic_id`, `learner_id`, `scenario_seed_index`, `started_at`. A new row is created by `POST /api/chat/start` and the server returns the `sessionId`. Persisting the scratchpad in the same row keeps the read pattern atomic.

**Alternative considered:** In-memory `ConcurrentHashMap<UUID, Scratchpad>` â€” rejected because it loses state on restart and complicates horizontal scaling. Redis â€” rejected because it introduces an operational dependency the project doesn't otherwise need; can be added later if needed for low-latency reads.

### D5. **`AiService.streamGenerate` interface evolves** to take a `List<ChatMessage>` instead of a flat user string.

**Why:** All three providers (Gemini, OpenAI, Ollama) natively accept alternating `messages[]`. The flattening was a downgrade â€” keeping it costs Leak A from the audit. Updating all three implementations in lockstep is the only way to fix this. Since `AiService` is an internal interface (no external consumer), this is a low-blast-radius breaking change.

**Alternative considered:** Add a separate `streamGenerateMulti` method and keep both â€” rejected because dead-code accumulates; once we've moved chat to multi-turn we never want to use the flat path again.

### D6. Curated scenario seeds + curated opening lines, **deterministic per topic**, replacing model-generated scenarios.

**Why:** Determinism is non-negotiable for the eval harness (Layer 3 goldens); without it, the same golden conversation produces different scratchpad states across runs. Curated seeds also fix the "scenario text was generated by the same weak model" compounding risk. v1 ships one seed and one opening per topic; later variants (3+ each) are a small, additive follow-up.

**Alternative considered:** Generate once and cache per learner â€” rejected, still non-deterministic across users and still unauditable. Generate but pin a seed â€” rejected, Ollama doesn't reliably honor seeds across model versions.

### D7. **3-layer eval harness** split across JVM (L1, L2) and Python (L3).

**Why:** L1/L2 are pure-function and mock tests that belong next to the code they protect â€” they run on every commit. L3 is slow (~2-5 min) and probabilistic; Python is the natural environment for prompt evals, and `backend-python` already runs the Ollama bridge. L3 reaches the assembled prompt via a dev-only debug endpoint to avoid duplicating prompt-assembly logic.

**Alternative considered:** Pure JVM evals â€” rejected because Java is a poor fit for iterating prompts. Pure Python evals â€” rejected because it duplicates assembly logic that lives in Java. The hybrid keeps each layer in its natural home.

### D8. Suggestions become **per-topic policy** (`ON`/`OFF`/`HINT`), tutor overlay becomes **per-topic policy** (`SUBTLE_RECAST`/`PAREN`/`OFF`).

**Why:** The audit showed asymmetric scenarios (job interview, doctor) are actively harmed by suggestions and inline corrections. The persona card carries the policy; the prompt template branches on it; the frontend respects `SUGGEST=OFF` by hiding the chip strip. Default policies cluster by topic type (transactional â†’ `SUBTLE_RECAST`/`ON`, asymmetric â†’ `OFF`/`OFF-with-HINT`, free-form â†’ `SUBTLE_RECAST`/`ON`). Individual topics override.

### D9. Streaming-UX defect is **acknowledged but deferred**.

**Why:** [ChatView.vue](../../../frontend-vue/src/views/ChatView.vue) currently shows raw JSON characters during the stream (the parser only runs at end). Fixing this would require either provider-native structured output mode (per-provider work) or a different output shape (natural reply first, structured tail separately). Both are non-trivial and would expand the change beyond a tractable scope. Calling it out so we don't claim it's fixed.

## Risks / Trade-offs

- **[gemma2:9b can't reliably emit the scratchpad shape for asymmetric topics]** â†’ mitigated by D1 (second-pass extraction uses a focused prompt + low temperature on a smaller model; falls back to the previous turn's scratchpad on parse failure). Tracked in eval Layer 3.
- **[extraction call adds latency, possibly visible if learner types fast]** â†’ mitigated by running extraction in parallel with the user reading the streamed reply; race-condition policy is "use previous scratchpad if N+1 arrives before N's extraction completes." A stale-rate metric is logged. If stale-rate exceeds a threshold (e.g., 10%) we switch to synchronous extraction.
- **[Ollama RAM budget when running two models concurrently]** â†’ must verify on deployment target. If insufficient, fall back to scratchpad-by-same-model (degrades to a one-model setup with extraction as a separate call to the same model).
- **[Persona cards drift from the actual user experience over time]** â†’ mitigated by Layer 3 goldens. The eval becomes the contract.
- **[Tasks list is large; risk of partial landing]** â†’ mitigated by phasing: Phase 1 lands the multi-turn API + persona cards without scratchpad (the prompt audit improvements alone are valuable). Phase 2 adds the scratchpad + chat_session. Phase 3 adds the eval harness. Each phase is independently shippable; the design is composed so Phase 2 and Phase 3 don't require redoing Phase 1.
- **[New `chat_session` table conflicts with the in-flight `docker-deployment` change]** â†’ mitigated by sequencing: docker-deployment lands first (already 74/94 tasks). Database migration for `chat_session` runs after.
- **[Per-topic authoring quality varies]** â†’ mitigated by L1 persona-card invariants (every card MUST have all fields populated; DOESN'T list MUST be non-empty; vocab anchors MUST be present at A2 minimum).

## Migration Plan

**Phase 1 â€” Persona cards + multi-turn (no scratchpad yet).**
1. Add `TopicType`, `PersonaCard`, `ScenarioSeed` records to `TopicDto.java`.
2. Author all 12 persona cards (~1 day).
3. Extend `AiService` interface to `streamGenerate(systemPrompt, List<ChatMessage>)`. Update all three providers.
4. Replace `buildRoleplayPrompt` / `buildConversation` with `PromptAssembler` that emits a system prompt + messages array.
5. Replace model-generated scenarios with curated seeds in `startScenario`.
6. Add Layer 1 (prompt-assembly) tests for all 12 topics.
7. Ship. Chat is already meaningfully better at this point â€” no scratchpad needed yet.

**Phase 2 â€” Scratchpad + session persistence.**
8. Add `chat_session` JPA entity + repository + migration.
9. Add `ChatSessionService` and route `startScenario` and `reply` through it.
10. Add `ScratchpadExtractor` calling the smaller Ollama model (or fallback to same model).
11. Pull extraction model in `ollama/entrypoint.sh` (coordinate with `docker-deployment`).
12. Inject scratchpad into the system prompt; update `PromptAssembler` per topic type.
13. Frontend: thread `sessionId` through `startScenario` / `streamReply`.
14. Add Layer 2 mock-model tests for scratchpad update + stale fallback.

**Phase 3 â€” Eval harness.**
15. Add dev-only `POST /api/internal/debug/assemble-prompt` (guarded by `dev` profile).
16. Author three golden conversations (restaurant / interview / environment).
17. Build Python eval runner under `eval/`. Wire to nightly job and pre-merge for chat-touching branches.

**Rollback:** Phase 1 is reversible by reverting the commits and clients tolerate the additive `sessionId`. Phase 2 requires reverting the migration (the `chat_session` table can be left in place; the application code can ignore it). Phase 3 is purely additive (test infra); revert is a code-only rollback.

## Open Questions

- **Which small Ollama model do we use for extraction?** Candidates: `qwen2.5:1.5b` (fast, good JSON), `phi3:mini` (3.8B, higher quality, slower). A small spike during Phase 2 picks one based on extraction-quality on the three golden conversations. Until then, the design assumes "a small instruct-tuned model that Ollama can run alongside gemma2:9b."
- **Race-condition policy for fast-typing learners:** v1 ships "use previous scratchpad if N+1 arrives early" + a `stale_rate` log metric. Open question whether we need a synchronous-extraction fallback in v1 or can defer.
- **First-turn scratchpad policy:** confirmed `null` on T0 (no extraction call). For T1, do we extract from the learner's first message before generating T1's reply (synchronous), or extract after (next-turn-uses-it, lazy)? Lazy chosen for v1; revisit if it causes noticeable quality dip on transactional topics where the first message often has the most useful slots ("two please" â†’ party_size=2).
- **Frontend resume policy on refresh:** v1 does not restore mid-session. The `sessionId` is in memory only. Once `chat_session` exists, the implementation cost to add `localStorage.sessionId` is small; we could land it inside Phase 2 if cheap, otherwise defer.
- **Coordination with `docker-deployment`:** confirm sequencing with the in-flight change. The Ollama service config and entrypoint must accommodate pulling a second model at startup. If `docker-deployment` archives before Phase 2 begins, this is a clean modification; if not, both branches need to merge cleanly.
